# 매왕길드 관리 툴 — 기획 문서

> 서랍길드(`index.html`) 코드를 베이스로 **매왕길드**용 관리 툴을 만들기 위한 설계 메모.
> 같은 게임(서랍길드에 소스를 제공한 길드) — 컨텐츠 종류·점수 단위는 서랍길드와 동일, 점수 산정 방식만 다름.
> 아직 구현 전, 방향 확정용 문서.

## 1. 서랍길드 버전과의 핵심 차이

| 항목 | 서랍길드(현재) | 매왕길드 |
|---|---|---|
| 컨텐츠 종류 / 점수 단위 | 토벌전·대항전·길드보스·수련장 / 억·만 | **동일** (같은 게임) |
| 점수 산정 | 투력 구간별 **고정 기준치** 테이블과 실제 점수를 비교 (기준치 페이지에서 구간/점수 설정) | 기준치 테이블 없음. **투력순위 vs 컨텐츠순위**를 비교 |
| 페이지 | 현황판 / 결과입력 / 길드원 / 기준치 / 정보 (5개) | 현황판 / 결과입력 / 길드원 (3개) — 기준치·정보 페이지 없음 |
| MVP 정의 | 기준치 대비 초과분(diff)이 가장 큰 사람 | 동일한 diff 개념으로 **유지** (아래 3장 참고) |
| Firebase | 기존 `seorap-guild` 프로젝트 | **신규 Firebase 프로젝트** 생성 |
| 관리자 비밀번호 | `0068` | `0318` |
| 초기 데이터 | 하드코딩 시드(`seed()`) | **매왕길드 엑셀 기록(수개월치)을 초기 DB로 변환 예정** — 파일 추후 첨부 |

## 2. 점수 로직 상세

- **투력순위**: 해당 대결의 `powerSnapshot`(대결 시점 전투력 스냅샷) 기준, 전체 길드원을 투력 내림차순 정렬했을 때의 순위.
- **컨텐츠순위**: 그 대결에서 입력한 `bossScores`(컨텐츠 점수) 기준, 참가자를 점수 내림차순 정렬했을 때의 순위.
- **diff = 투력순위 − 컨텐츠순위**
  - 투력 10위인데 컨텐츠 3위 → `+7` → 초록 (기대보다 잘함)
  - 투력 3위인데 컨텐츠 10위 → `-7` → 빨강 (기대보다 못함)
  - 0이면 회색(기대만큼)
- 이 계산은 이미 서랍길드 코드에 죽은 코드로 존재:
  - `pRank(name)` — 현재 투력 기준 순위 (index.html:762)
  - `score(resultRank, name)` = `pRank(name) - resultRank` (index.html:769)
  - `sc(s)` / `ss(s)` — 색상 클래스 / 부호 문자열 (index.html:770-771)
  - `scoreBySnap` — 스냅샷 기준 버전, 결과입력 페이지 안에 정의만 되어있고 렌더링에는 미사용 (index.html:954-963)
  - → 신규 버전에서는 이 함수들을 **되살려서** 대시보드 렌더링에 실제로 연결하면 됨.
- 기준치 페이지가 없으므로 `getStandards`, `getExpectedScore`, `S.standards`, 관련 UI(범위 추가 모달, `.std-*` CSS)는 전부 제거.
- `computeMvpCounts`(index.html:779-798)는 현재 `stdAvailable`/`getExpectedScore` 기반이라 그대로 못 씀 → **diff = pRank - resultRank 최댓값 기준**으로 재작성 필요 (뱃지 카운트/노출 UI는 유지).

## 3. 재사용 vs 제거

### 그대로 재사용
- 단일 HTML + Firebase Realtime Database 구조 (신규 길드용 Firebase 프로젝트 새로 생성 필요)
- 다크 테마 CSS 변수/컴포넌트 스타일
- 길드원 관리 페이지 (전투력 입력, 정렬, 직업뱃지, 추방 등) 로직 그대로
- 결과입력 페이지의 달력 모달로 대결 추가 (`openCalModal`/`calConfirm`), `powerSnapshot` 저장 방식
- `saveEntryScore`/`saveAllEntryScores`의 "점수 입력 → 컨텐츠순위(result 배열) 자동 재계산" 로직 — 그대로 유지 가능, 대시보드에서 쓰는 비교식만 바뀜
- 관리자 비밀번호 게이팅 구조 (값은 `0318`로 교체)
- 대결 타입 탭 구조(`BATTLE_TYPES`) — 같은 게임이라 값 그대로 유지 (토벌전/대항전/길드보스/수련장)
- MVP 배지(★) 로직(`computeMvpCounts`) — diff 계산 방식만 기준치 기반 → 순위 기반으로 바뀔 뿐, 카운트/뱃지 노출 방식은 그대로 유지

### 제거
- 기준치 페이지 전체: `renderStandardPage`, `getStandards`, `getExpectedScore`, 범위 추가 모달(`range-modal-*`), `.std-*` CSS
- 정보 페이지 전체: `renderInfoPage`, `INFO_TABS`, `infoTab`, `.info-*`/`.tip-*` CSS, `public/antique`·`public/boss_match` 이미지 에셋
- 대시보드의 "기준 초과/기준 미달" 칩, `stdAvailable` 분기 → "투력순위/컨텐츠순위 diff" 테이블로 교체
- 이미 죽어있던 구식 슬롯 선택 모달(`openModal`/`pick`/`renderModalList`, `.slot`/`.mitem` 등)도 신규 버전에 옮길 필요 없음 (애초에 서랍길드에서도 안 씀)

## 4. 데이터 모델 (신규 버전)

```
S = {
  members: [{name, job, power, powerDate, prevPower, prevPowerDate}],
  battles: [{
    id, date, type,
    result: [...],        // 점수순 정렬된 이름 배열 (자동 계산, 유지)
    bossScores: {name: score},
    powerSnapshot: {name: power}   // 필수 — 투력순위 계산 기준
  }]
  // standards 필드 없음
}
```

## 5. 확정된 사항

1. 컨텐츠 종류: 서랍길드와 동일 (토벌전/대항전/길드보스/수련장)
2. 점수 단위: 서랍길드와 동일 (억/만)
3. Firebase: **매왕길드 전용 신규 프로젝트** 생성 (서랍길드 DB와 완전 분리)
4. 관리자 비밀번호: `0318`
5. MVP 배지(★): 유지, 단 판정 로직은 diff 기반으로 재작성 (2장 참고)

## 6. 초기 데이터 — 엑셀 시딩

- 매왕길드가 수개월간 기록해 온 엑셀 파일을 추후 첨부 예정 → 서랍길드 `seed()`(index.html:630-702)처럼 초기 데이터를 하드코딩하는 대신, **엑셀 내용을 그대로 변환**해서 최초 시드 데이터로 사용.
- 엑셀 첨부되면 확인할 것:
  - 길드원별 전투력 이력(날짜별) → `members[].power/prevPower/prevPowerDate` 매핑 가능한지
  - 대결(날짜/타입)별 참가자 점수 이력 → `battles[].bossScores`로 매핑 가능한지
  - 각 대결 시점의 전투력 스냅샷이 엑셀에 남아있는지 (없으면 `powerSnapshot`은 그 시점 투력을 역산하거나 최신 투력으로 대체해야 함 — 정확도 이슈 있을 수 있음, 파일 보고 판단)
- 엑셀 → `seed()` 형태 JS 객체 변환은 파일 받은 뒤 별도 스크립트로 처리 예정.
